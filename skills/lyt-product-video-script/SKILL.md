# lyt-product-video-script：产品短视频拍摄脚本

本 skill 把已确认的产品事实翻译成目标用户能看见、能理解的购买理由，再将购买理由组织成可拍摄、可执行、可验证的电商短视频脚本。

脚本是内容假设，不是爆款预测或成交承诺。

## 通用规则

进入本 skill 后，先读取并遵守 `../lyt/references/common-rules.md`。本文件只补充产品短视频脚本的专属边界、证据闸门和执行流程；如有冲突，以更严格的事实、合规和可执行性约束为准。

## 使用边界

只处理：

- 根据已有产品、图片、链接、资料或卖点生成一份完整拍摄脚本
- 把已有产品文案、口播稿或内容构想整理成可拍分镜
- 生成真人出镜、只露手或无人物的产品视频脚本
- 生成自然流量、商品点击、广告素材或达人寄样场景的脚本
- 生成目标市场语言的台词、屏幕文字和中文回译
- 用户明确要求时，为同一产品生成多个单变量测试角度

不处理：

- 从 0 推荐类目或产品，转 `lyt-product-selection`
- 判断产品能不能测、能不能做，转 `lyt-product-validation`
- 诊断商品主图、详情页或商品首屏，转 `lyt-product-page-diagnosis`
- 分析已发布视频的播放、完播、点击、加购或下单数据，转 `lyt-data-analysis`
- 直接生成、拍摄或剪辑视频
- 非商品类的个人 IP、知识口播和泛自媒体文案

用户不需要先完成产品验证才能调用本 skill。只要已经有具体产品且目标是生成拍摄脚本，就直接进入；但不得因脚本可生成就推断产品值得做。

## 支持输入

- 产品图片、商品截图、商品链接、供应链资料或产品说明
- 已确认的功能、材质、规格、卖点、人群和使用场景
- 国家/地区、平台、目标语言和视频目标
- 现有产品文案、口播稿、分镜草稿或拍摄想法
- 拍摄条件：人物、场地、设备、时长、道具、是否露脸和后期能力
- 目标风格：UGC、使用演示、痛点解决、对比、测评、场景、细节证明或其他风格

不修改原始附件。产品名、SKU、货号、品牌、型号、规格、链接和目标市场等原始标识保留原文。

## 什么时候读取知识包

除越界转交外，进入产品短视频脚本场景后读取 `references/product-video-script.md`。

用户要求多版脚本、A/B 测试、测试指标或继续/暂停条件时，再读取 `../lyt/references/experiment-rules.md`。

## 基本公理

1. **事实公理：** 脚本只能使用已确认的产品信息，不虚构功能、材质、认证、评价、销量和使用效果。
2. **受众公理：** 产品卖点只有进入具体人群和使用场景，才可能转化为购买理由。
3. **视觉公理：** 短视频首先是可见的动作、过程、变化和结果，不是把商品介绍改写成口播。
4. **单一命题公理：** 一条短视频优先验证一个核心购买理由，不同时堆叠全部卖点。
5. **一致性公理：** 开头吸引的人、正文展示的问题、产品提供的解决方式和最终行动引导必须一致。
6. **可执行公理：** 无法在用户现有人员、场地、设备、预算和时间内完成拍摄的脚本，不是有效脚本。
7. **可验证公理：** 脚本是对内容效果的待验证假设，不是流量或成交保证。

## 决策原则

1. 先提取产品事实，再决定内容角度。
2. 先确定视频目标，再设计开头。
3. 按「人群 → 场景 → 问题 → 产品动作 → 可见结果」组织内容。
4. 产品卖点必须转化为镜头中的动作、细节、对比或证据。
5. 没有证据支持的功效和承诺不能进入台词或屏幕文字。
6. 口播不能替代产品演示，屏幕文字不能替代关键镜头。
7. 开头不仅要制造停留，还必须与产品购买理由相关。
8. 一条脚本只保留一个主要内容机制。
9. 拍摄条件是脚本生成前的约束，不是生成后的附加检查。
10. 批量测试不同角度时，每条视频只改变一个主要变量。

## 首轮响应闸门

1. 没有具体产品时，先要求用户提供产品图片、链接、商品名或资料，不自行选品。
2. 有产品资料时，先提取「可见事实 / 用户陈述 / 推断 / 未知」，不把外观猜测写成功能事实。
3. 只追问最影响脚本的关键缺口。优先级为：目标市场/平台 → 视频目标 → 拍摄条件 → 目标人群/场景 → 风格。
4. 风格未说明且会改变拍法时，询问用户要 UGC、使用演示、痛点解决、对比、测评、场景、细节证明还是自定义风格。
5. 用户明确要求「直接生成」或把选择权交给 Agent 时，可以做最小假设后生成，但必须在脚本简报中显式写出假设。

## 执行流程

1. 识别任务模式：从产品生成、已有文案转分镜、目标语言本地化或多角度测试。
2. 提取产品事实，标记未知信息和表达风险。
3. 确定国家/地区、平台、目标语言、视频目标和拍摄条件。
4. 建立「人群—场景—需求—产品解决方式」假设。
5. 选择一种主要内容机制，提炼一个核心购买理由。
6. 将核心购买理转化为可见的镜头动作、台词、屏幕文字和声音。
7. 按用户的人物、场地、设备、道具、时长和后期能力收缩拍摄方案。
8. 检查开头、人群、场景、产品动作、购买理由和 CTA 是否一致。
9. 检查功效、认证、价格、优惠、评价、对比和本地化表达风险。
10. 输出一份可直接拍摄的脚本，说明本条视频正在验证的内容假设。

## 输出要求

默认只生成一份完整脚本。只有用户明确要求批量、多版、A/B 或多角度测试时，才生成多份脚本。

| 档位 | 适用场景 | 输出 |
| --- | --- | --- |
| 轻量 | 产品或关键条件不足 | 事实提取 + 一个关键缺口 + 下一步 |
| 标准 | 产品、目标和拍摄条件足够 | 脚本简报 + 分镜表 + 拍摄清单 + 风险 + 验证假设 |
| 测试组 | 用户明确要多角度测试 | 共享基线 + 多份单变量脚本 + 测试卡 |

标准输出结构：

```markdown
# 产品短视频脚本：{产品名}

## 产品事实与待确认信息
{已确认事实、用户陈述、未知信息和不得使用的表达}

## 脚本简报
{国家/平台、目标语言、视频目标、人群、场景、风格、时长、拍摄条件、核心购买理由}

## 拍摄脚本
| 时间 | 画面与动作 | 景别/机位 | 口播或声音 | 屏幕文字 | 产品信息 | 剪辑提示 |
| --- | --- | --- | --- | --- | --- | --- |

## 拍摄清单
{产品、道具、人物、场地和需要预先准备的素材}

## 事实与合规风险
{不得讲、待确认、需替换的表达}

## 本条验证什么
{一个内容假设 + 对应的用户行为}
```

## 转交规则

| 触发条件 | 下一步 |
| --- | --- |
| 用户没有具体产品，想先找方向 | 转 `lyt-product-selection` |
| 用户真正想判断产品能不能测 | 转 `lyt-product-validation` |
| 用户要诊断主图、详情页或商品首屏 | 转 `lyt-product-page-diagnosis` |
| 用户提供已发布视频的播放、完播、点击、加购或下单数据 | 转 `lyt-data-analysis` |
| 用户只说视频没流量、不出单，但对象、动作和数据不清 | 转 `lyt-problem-clarifier` |

转交时保留产品、国家/平台、已确认事实、脚本假设、表达风险和最关键缺口。

## 禁止事项

- 不虚构产品功能、材质、尺寸、认证、测试结果、用户评价、销量、折扣或稀缺性。
- 不从产品外观猜测功效、人群、材质或适用范围。
- 不用「必爆」「稳卖」「一定成交」「平台一定给流量」描述脚本。
- 不把所有卖点、内容机制和目标人群堆进同一条视频。
- 不只写口播，必须说明可拍画面、动作、道具和镜头。
- 不在用户只要一份脚本时自动生成多份脚本。
- 不照抄竞品台词、品牌身份、用户评价或受保护的创意表达。
- 不因为能生成脚本，就判断产品值得做或能够赚钱。
